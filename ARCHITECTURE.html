🥗 FoodFlow v3 — How It Works

A plain-English walkthrough. No prior knowledge assumed.

What is this app?

FoodFlow is a meal-planning web app. A user logs in, describes recipes to an AI assistant in plain conversation, and the app saves them. Those recipes then show up in a weekly meal grid. That's the whole thing.

Here are the four things it does:

1. Login
Email + password, managed by Firebase. Sessions persist across browser reloads.
2. Add recipes via AI
Chat with an AI assistant that asks questions and extracts a structured recipe.
3. Store recipes
Each user's recipes are saved to Firebase under their unique user ID.
4. Week planner
A 7-day grid showing the user's saved recipes. Meal slots are a work-in-progress.

What is React? (and why does it matter)

React is the core framework this app is built on. Before you read the code, you need to understand three things React does. Everything else follows from these.

1. Components are just functions that return HTML

In React, every piece of the UI is a component — a JavaScript function that describes what to show on screen. Here's the simplest possible example:

function Greeting() { return <p>Hello, world!</p> }

That <p> inside a JS function looks weird — it's called JSX. It's not real HTML, it's a shorthand that compiles down to normal JavaScript. React turns it into actual DOM elements in the browser.

Analogy: Think of components like LEGO bricks. Each brick knows how to draw itself. You combine bricks to build the full page. App is the base plate. AppShell, WeekPlanner, RecipeChat are bricks stacked on top of it.

2. State — variables that trigger redraws

A normal JavaScript variable changing doesn't update the screen. React's useState hook creates a special variable: when you change it, React automatically re-runs the component function and updates what's shown.

// From App.tsx — the actual code in this project const [user, setUser] = useState(null) // user → the current value (null = not logged in) // setUser → the function to change it // When you call setUser(someFirebaseUser), React immediately // re-renders App and shows AppShell instead of LoginPage.
Analogy: useState is like a sticky note on a whiteboard. When you erase and rewrite the note (setUser), everyone looking at the whiteboard (the UI) sees the new value instantly.

3. Effects — code that runs when things change

useEffect lets you run code at the right moment — usually when a component first appears on screen, or when a specific value changes.

// From App.tsx — the actual code in this project useEffect(() => { // This runs once when App first loads. // It tells Firebase: "call me whenever login state changes". const unsubscribe = onAuthChange((firebaseUser) => { setUser(firebaseUser) // updates state → re-renders App setLoading(false) }) return unsubscribe // cleanup: stop listening when App unmounts }, []) // ← the [] means "only run once, on first render"
Analogy: useEffect is like saying "when this component appears, do X". The function you return is the cleanup — "when this component disappears, do Y". The empty [] means "only do this once".

4. Props — passing data down to child components

Components can receive data from their parent, called props. It works like function arguments.

// AppShell.tsx receives the logged-in user as a prop export default function AppShell({ user }: Props) { // user.uid → the unique ID Firebase assigned to this user // user.email → their email address (shown in the top nav) // AppShell then passes userId down to its children: return <WeekPlanner userId={user.uid} /> }

Data flows in one direction: parent → child. App knows the user. It passes user to AppShell. AppShell extracts user.uid and passes it to WeekPlanner and RecipeChat. Those components use the uid to read/write the right user's data in Firebase.

How the app boots up

When you open the app in a browser, exactly this happens — in order:

1
Browser downloads index.html. The whole body is just this: <div id="root"></div> — an empty container. Nothing visible yet.
↓
2
main.tsx runs. It finds that <div id="root"> and tells React: "take control of this div and put <App /> inside it".
↓
3
App.tsx renders. Its initial state is user = null and loading = true, so it shows a "Loading…" spinner.
↓
4
App.tsx's useEffect runs immediately. It calls onAuthChange() from firebase.ts, which asks Firebase: "is there a saved session from a previous visit?"
↓
5a
No session: Firebase calls back with null. setUser(null) + setLoading(false). App re-renders and shows <LoginPage />.
5b
Session found: Firebase calls back with the user object. setUser(firebaseUser) + setLoading(false). App re-renders and shows <AppShell user={user} /> — no login needed.

Key insight

You never manually redirect the user. Firebase fires the callback, React re-renders automatically. This is the same mechanism for login, logout, and page refresh — always the same callback.

// The full App.tsx — nothing hidden export default function App() { const [user, setUser] = useState<User | null>(null) const [loading, setLoading] = useState(true) useEffect(() => { const unsubscribe = onAuthChange((firebaseUser) => { setUser(firebaseUser) setLoading(false) }) return unsubscribe }, []) if (loading) return <div>Loading…</div> if (!user) return <LoginPage /> return <AppShell user={user} /> }

Login flow

When a user fills in the form in LoginPage.tsx and hits Submit:

1
LoginPage calls loginWithEmail(email, password) from firebase.ts
↓
2
Firebase checks the credentials. If correct, it saves a session cookie in the browser and returns a User object.
↓
3
Firebase automatically fires the onAuthChange callback that App.tsx set up earlier — passing in the new User object.
↓
4
App.tsx calls setUser(firebaseUser). React re-renders App, and now it shows <AppShell /> instead of <LoginPage />.

Logout works the same way in reverse. AppShell calls logout(), Firebase clears the session, onAuthChange fires with null, setUser(null) runs, and LoginPage shows up again.

Registration works identically — registerWithEmail(email, password) creates the account and immediately fires onAuthChange with the new user. No separate login step needed.

Recipes & live data

This is the most interesting part of the data architecture. Recipes don't load once — they stay live. Here's what that means.

How WeekPlanner loads recipes

1
WeekPlanner mounts and its useEffect runs. It calls subscribeToRecipes(userId, setRecipes).
↓
2
Inside firebase.ts, this calls onValue(ref(db, 'users/{userId}/recipes'), callback). This opens a persistent connection to Firebase — like a phone call that stays on the line.
↓
3
Firebase immediately fires the callback with the current recipe list. setRecipes(list) runs → WeekPlanner re-renders → recipes appear on screen.
↓
4
The connection stays open. Any time data changes in Firebase (a new recipe is added from RecipeChat), Firebase fires the callback again automatically with the updated list.
Why this matters: You don't need a refresh button. If you add a recipe via the AI chat on one tab, WeekPlanner on the other tab updates instantly — because Firebase called setRecipes again without you asking.
// From firebase.ts — the actual subscribe function export const subscribeToRecipes = ( userId: string, callback: (recipes: Recipe[]) => void ) => { const recipesRef = ref(db, `users/${userId}/recipes`) // onValue = "every time this path changes, call me" return onValue(recipesRef, (snapshot) => { if (!snapshot.exists()) return callback([]) const raw = snapshot.val() const list = Object.entries(raw).map(([id, data]) => ({ id, ...data })) callback(list) }) // The return value is an "unsubscribe" function. // WeekPlanner calls it on cleanup to close the connection. }

Saving a new recipe

// From firebase.ts — saveRecipe export const saveRecipe = async (userId, recipe) => { // push() creates a new item under users/{userId}/recipes // with a Firebase-generated unique key like "-OHgt3kx..." const recipeRef = push(ref(db, `users/${userId}/recipes`)) await set(recipeRef, { ...recipe, createdAt: new Date().toISOString() }) return recipeRef.key // returns the new ID so we can reference it }

After saveRecipe runs, Firebase writes to users/{userId}/recipes. This triggers the live listener in WeekPlanner — which fires setRecipes — which causes a re-render. The new recipe card appears instantly without any manual refresh.

The AI chat trick (how recipes are extracted)

This is the cleverest part of the app. The AI doesn't have a special "save recipe" button — it embeds the recipe inside its normal text response using a secret tag.

How it works

1
User types a message. RecipeChat calls chatAboutRecipes() in openrouter.ts, sending the full conversation history + a system prompt.
↓
2
The system prompt tells the AI: "Once you have all the recipe details, put the final recipe at the end of your response like this: RECIPE_JSON: {...}"
↓
3
The AI responds conversationally — "Great! I'll set servings to 4. Here's your recipe: 📋 Draft: ..." — and when ready, appends the hidden JSON at the end.
↓
4
RecipeChat scans the AI's response with a regex: /RECIPE_JSON:\s*(\{.+\})/. If it finds a match, it parses the JSON and calls saveRecipe().
↓
5
The RECIPE_JSON: {...} block is stripped out of the displayed message. The user only sees the friendly text, not the raw JSON. A green "✅ Saved" card appears at the bottom of the message.
// From RecipeChat.tsx — the detection logic const response = await chatAboutRecipes(...) // Look for RECIPE_JSON: followed by a JSON object const jsonMatch = response.match(/RECIPE_JSON:\s*(\{.+\})/s) if (jsonMatch) { const parsed = JSON.parse(jsonMatch[1]) // parse the recipe object const savedId = await saveRecipe(userId, parsed) // save to Firebase detectedRecipe = { id: savedId, ...parsed } } // Strip the JSON block before showing the message to the user const displayText = response.replace(/RECIPE_JSON:\s*\{.+\}/s, '').trim()

The system prompt is what controls the AI

In openrouter.ts, the big string at the top of chatAboutRecipes() is the system prompt. It tells the AI who it is, what fields to collect, how to format the output, and when to emit the RECIPE_JSON block. If you want to change how the AI behaves — ask different questions, add more fields, change its tone — edit that string.

How the AI "remembers" the conversation

The AI has no memory between calls. Every time the user sends a message, RecipeChat sends the full conversation history — every message from the beginning — to the API. The AI reads all of it and responds as if it remembers. This is how all chatbots work.

// From openrouter.ts — the messages array that gets sent every time chat([ { role: 'system', content: systemPrompt }, // AI instructions { role: 'assistant', content: "Hi! What recipe..." }, // message 1 { role: 'user', content: "Pasta carbonara" }, // message 2 { role: 'assistant', content: "How many servings?" }, // message 3 { role: 'user', content: "4" }, // message 4 (new) ])

Database structure

Firebase Realtime Database stores data as a JSON tree. Each user gets their own subtree, identified by their uid — a unique ID Firebase generates when they register. No user can read another user's data.

users/ └── {uid} ← e.g. "abc123xyz" — Firebase generates this on register │ ├── recipes/ │ └── {recipeId} ← Firebase generates this key via push() │ ├── name: "Pasta Carbonara" │ ├── prepTime: 10 ← minutes │ ├── cookTime: 20 │ ├── servings: 2 │ ├── emoji: "🍝" │ ├── ingredients: [ {name: "egg", quantity: "2"}, ... ] │ ├── instructions: [ "Boil pasta", "Fry guanciale", ... ] │ ├── tags: ["dinner", "italian"] │ └── createdAt: "2026-03-31T12:00:00Z" │ ├── goals/ │ ├── healthyEating: true │ ├── weightLoss: false │ ├── muscleGain: true │ └── budgetFriendly: false │ └── mealPlans/ └── "2026-03-31" ← the Monday of that week ├── monday: { breakfast: "{recipeId}", lunch: null, dinner: "{recipeId}" } ├── tuesday: { breakfast: null, lunch: "{recipeId}", dinner: null } └── ...

Meal plans store recipe IDs, not the full recipe. When the week planner grid displays a meal, it will look up the recipe by its ID from the recipes list. This way if a recipe is updated, the meal plan reflects the change automatically.

Rule of thumb

Changing how data is stored or fetched → edit firebase.ts
Changing AI behaviour → edit the system prompt in openrouter.ts
Changing what the user sees → edit a file in components/
Adding a new field to a recipe → edit types.ts first, then firebase.ts if the DB path changes
